- Q: 실시간 갱신 대상 기준을 어떻게 정의할까? → A: 유료 구독 활성 또는 고레벨 점수(70점 이상) 충족 시 실시간 갱신 제공한다.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 예측 대시보드 사용 (우선순위: P1)

개미 투자자는 모바일 웹뷰에서 종목별 AI 예측값과 핵심 지표를 확인해 다음 거래일 전략을 세울 수 있어야 한다.

**우선순위 이유**: 핵심 가치인 예측 결과 확인이 실패하면 서비스 목적이 성립하지 않는다.

**독립 테스트**: 비로그인 또는 로그인 상태에서 대시보드에 진입해 종목 선택, 예측값 확인, 신뢰도 확인까지 완료하면 독립 검증 가능하다.

**인수 시나리오**:

1. **조건** 사용자가 로그인된 상태다, **행동** 대시보드에서 종목을 선택한다, **결과** D+1~D+5 예측값과 신뢰도가 표시된다.
2. **조건** 사용자가 다른 종목으로 전환한다, **행동** 대시보드를 새로 조회한다, **결과** 해당 종목 기준 데이터로 즉시 갱신된다.

---

### 사용자 스토리 2 - 예측 근거와 성능 이력 검토 (우선순위: P2)

사용자는 예측 근거 지표와 과거 정확도/초과 수익률 이력을 비교해 모델 신뢰도를 판단할 수 있어야 한다.

**우선순위 이유**: 근거와 성능 이력이 없으면 예측값 신뢰가 낮아지고 재방문율이 떨어진다.

**독립 테스트**: 비로그인 또는 로그인 상태에서 근거 탭과 이력 탭의 지표 카드, 스파크라인, 성능 요약을 확인하면 독립 검증 가능하다.

**인수 시나리오**:

1. **조건** 사용자가 근거 탭에 있다, **행동** 핵심 지표를 본다, **결과** 예측에 사용된 지표값과 상태를 확인할 수 있다.
2. **조건** 사용자가 이력 탭에 있다, **행동** 기간을 바꿔 조회한다, **결과** 과거 예측과 실제 결과 비교가 반영된다.

---

### 사용자 스토리 3 - 관심 종목 및 알림 관리 (우선순위: P3)

사용자는 관심 종목을 등록하고 알림 조건을 설정해 필요한 시점에 예측 업데이트를 받을 수 있어야 한다.

**우선순위 이유**: 개인화 기능은 장기 사용성과 유료 전환 가능성을 높인다.

**독립 테스트**: 관심 종목 추가/삭제, 알림 설정 저장/변경, 재접속 후 설정 유지가 확인되면 독립 검증 가능하다.

**인수 시나리오**:

1. **조건** 사용자가 종목 상세를 보고 있다, **행동** 관심 종목으로 추가하고 알림을 켠다, **결과** 개인 목록에 등록되고 알림 상태가 저장된다.
2. **조건** 사용자가 알림을 끈다, **행동** 저장 후 재접속한다, **결과** 알림 비활성 상태가 유지된다.

---

### 엣지 케이스

- 알림 배치 실패 시 실패 원인과 재시도 예정 상태를 사용자에게 표시해야 한다.
- 외부 데이터 지연 시 최신 예측 대신 마지막 유효 데이터와 갱신 시각을 보여줘야 한다.
- 유료 또는 고레벨 사용자에게 실시간 갱신을 제공하지 못하는 경우 등급별 기대 주기와 실제 주기 차이를 명확히 안내해야 한다.
- 예측 데이터가 없는 신규 종목은 빈 상태 안내와 대체 행동(다른 종목 선택)을 제공해야 한다.
- 네트워크 오류/타임아웃 시 사용자에게 재시도 경로를 제공해야 한다.
- 권한 없는 접근(타 사용자 관심 종목/설정 조회)은 거부되어야 한다.
- 비로그인 사용자가 관심 종목/알림 기능에 접근하면 로그인 유도 후 접근이 제한되어야 한다.
- 인앱 알림 확인 전까지 미확인 상태를 유지하고, 확인 시 읽음 처리되어야 한다.
- 짧은 시간 내 과도한 조회 요청은 제한되고, 제한 사유와 재시도 가능 시각을 안내해야 한다.
- 관심 종목/알림 설정 동시 수정 시 마지막 저장 충돌을 감지하고 사용자에게 재확인을 요구해야 한다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 종목별 단기 예측 결과(D+1~D+5)를 제공해야 한다.
- **FR-002**: 시스템은 예측 결과와 함께 신뢰도 지표를 제공해야 한다.
- **FR-003**: 시스템은 예측 근거 지표(가격/거래량/시장심리/기술 지표)를 제공해야 한다.
- **FR-004**: 시스템은 과거 예측과 실제 결과를 비교할 수 있는 이력 화면을 제공해야 한다.
- **FR-005**: 사용자는 관심 종목을 등록, 조회, 삭제할 수 있어야 한다.
- **FR-006**: 사용자는 종목별 알림 사용 여부와 조건을 저장/수정할 수 있어야 한다.
- **FR-007**: 시스템은 데이터 상태(정상/지연/실패)와 마지막 갱신 시각을 표시해야 한다.
- **FR-008**: 시스템은 데이터 요청 실패 시 오류 유형(네트워크, 권한, 데이터지연)을 구분한 메시지와 재시도 동작을 제공해야 한다.
- **FR-009**: 시스템은 사용자별 데이터 접근 권한을 분리해 타 사용자 데이터 노출을 방지해야 한다.
- **FR-010**: 시스템은 비용 최적화를 위해 주요 데이터/요약 결과를 정책 기반 캐싱으로 제공해야 한다.
- **FR-011**: 시스템은 모델 성능 지표(DA, MAPE, RMSE)를 사용자 또는 운영자가 확인 가능한 형태로 제공해야 한다.
- **FR-012**: 시스템은 GPS 및 정밀 위치 식별자를 저장하지 않아야 한다.
- **FR-013**: 시스템은 하루 1회 예측 배치 완료 후 알림 조건 충족 종목에 대해서만 알림을 발송해야 한다.
- **FR-014**: 시스템은 동일 종목/동일 일자에 대해 중복 알림을 발송하지 않아야 한다.
- **FR-015**: 시스템은 대시보드/근거/이력 조회를 비로그인 사용자에게 허용해야 한다.
- **FR-016**: 시스템은 관심 종목 및 알림 설정 기능을 로그인 사용자에게만 허용해야 한다.
- **FR-017**: 시스템은 알림 채널을 인앱 알림으로만 제공해야 한다.
- **FR-018**: 시스템은 사용자별 인앱 알림의 읽음/미읽음 상태를 저장하고 조회할 수 있어야 한다.
- **FR-019**: 시스템은 일반 사용자에게 장중 15분 주기 갱신과 장마감 후 1회 확정 갱신을 제공해야 한다.
- **FR-020**: 시스템은 유료 구독 활성 사용자 또는 고레벨 점수 70점 이상 사용자(둘 중 하나 충족)에게 1분 주기 실시간 갱신을 제공해야 한다.
- **FR-021**: 시스템은 사용자 등급별 갱신 정책 요약(일반 15분/프리미엄 1분)을 UI에 표시해야 한다.
- **FR-022**: 시스템은 실시간 갱신 권한 판정 근거(구독 활성 여부, 레벨 점수)를 사용자에게 확인 가능한 형태로 제공해야 한다.
- **FR-023**: 시스템은 캐싱 정책(적용 여부, 마지막 갱신 시각, TTL 상태)을 사용자에게 표시해야 한다.
- **FR-024**: 시스템은 사용자/세션 단위 요청 제한 정책을 적용하고 제한 초과 시 차단해야 한다.
- **FR-025**: 시스템은 관심 종목 및 알림 설정 저장 시 동시성 충돌을 감지하고 충돌 해결 절차를 제공해야 한다.

### 핵심 엔터티 *(기능이 데이터를 다룰 때 포함)*

- **사용자(User)**: 계정 식별자, 표시명, 개인화 설정을 가진다.
- **종목(Symbol)**: 티커, 시장, 표시명, 활성 상태를 가진다.
- **예측 결과(Prediction)**: 예측일, 예측값, 예측 방향, 신뢰도, 생성 시각을 가진다.
- **지표 스냅샷(IndicatorSnapshot)**: 예측 생성 시점의 주요 지표 집합을 가진다.
- **성능 기록(PerformanceRecord)**: DA, MAPE, RMSE 및 기준 기간 정보를 가진다.
- **관심 종목 항목(WatchlistItem)**: 사용자-종목 매핑과 등록/변경 시각을 가진다.
- **알림 설정(AlertPreference)**: 사용자별 종목 알림 여부, 임계 조건, 발송 상태를 가진다.
- **인앱 알림(AppNotification)**: 사용자별 알림 메시지, 생성 시각, 읽음 상태를 가진다.

## 헌법 정합성 *(필수)*

- **CA-001 개인정보**: GPS 또는 정밀 위치 식별자를 저장하지 않음을 보장한다.
- **CA-002 네이밍**: 구현 시 컴포넌트 파일명은 PascalCase, 변수명은 camelCase를 사용한다.
- **CA-003 품질 게이트**: `pnpm lint`, `pnpm typecheck` 통과를 완료 기준에 포함한다.
- **CA-004 추적성**: 단계별 `docs/개발일지/`, `docs/prompt/` 업데이트를 수행한다.

## 가정

- 웹뷰 컨테이너와 일반 브라우저에서 동일 핵심 기능 흐름을 제공한다.
- 인증 체계와 기본 사용자 프로필은 기존 프로젝트 기준을 사용한다.
- 데이터 소스(yfinance 등) 지연 가능성이 있으며, 지연 시 대체 표시가 허용된다.
- 예측 모델은 주기적으로 재학습/갱신되며, 성능 지표는 버전별로 관리된다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 비로그인 사용자 95% 이상이 30초 이내 첫 예측 결과를 확인한다.
- **SC-002**: 대시보드/근거/이력 핵심 탭 이동 성공률이 99% 이상이다.
- **SC-003**: 관심 종목 추가/삭제/알림 저장 작업의 완료율이 95% 이상이다.
- **SC-004**: 데이터 지연/오류 상황에서 상태 메시지 노출률이 100%다.
- **SC-005**: 권한 위반으로 인한 타 사용자 데이터 노출 사고가 0건이다.
- **SC-006**: 핵심 성능 지표 화면 조회 성공률이 99% 이상이다.
- **SC-007**: 일반 사용자 데이터 갱신이 공지된 주기(장중 15분, 장마감 후 1회) 기준 충족률 95% 이상이어야 한다.
- **SC-008**: 유료 또는 고레벨 사용자 실시간(1분 주기) 갱신 충족률이 95% 이상이어야 한다.
- **SC-009**: 로그인 사용자 95% 이상이 30초 이내 첫 예측 결과를 확인한다.
