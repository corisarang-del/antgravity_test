# Phase 0 Research - Ant Gravity AI 주가 예측 웹앱

## 1) 디자인 방향 (신뢰 + 퍼니 + 친근 + 깔끔)
- Decision: 베이스는 신뢰형 금융 UI(명확한 정보 위계, 절제된 색상)로 두고,
  마이크로 카피/아이콘/미세 모션에서만 퍼니 톤을 제한적으로 사용한다.
- Rationale: 투자 데이터 서비스는 신뢰 손상이 치명적이므로 기본 톤은 안정적이어야 하며,
  친근함은 보조 레이어로 적용해야 정보 신뢰성과 재미를 동시에 확보할 수 있다.
- Alternatives considered:
  - 강한 캐주얼/게임형 톤 전면 적용: 정보 신뢰 하락 위험으로 기각
  - 완전 보수적 금융 톤: 차별성과 사용자 친밀감 부족으로 기각

## 2) 갱신/알림 운영 정책
- Decision: 일반 사용자는 장중 15분 + 장마감 1회 확정 갱신, 유료 또는 고레벨은 1분 실시간.
  알림 채널은 인앱 전용, 하루 1회 배치 발송.
- Rationale: 데이터 소스/비용 제약을 고려할 때 계층형 갱신 정책이 가장 현실적이며,
  인앱 알림은 개인정보/운영 복잡도를 낮추고 웹뷰 환경과도 정합성이 높다.
- Alternatives considered:
  - 전 사용자 실시간: 비용/안정성 리스크 큼
  - 이메일/푸시 병행: 동의/전송 실패/운영 복잡도 증가

## 3) 인증/권한 모델
- Decision: 대시보드·근거·이력은 비로그인 조회 허용, 관심종목·알림 설정은 로그인 필수.
- Rationale: 핵심 탐색 진입 장벽을 낮추면서 개인화 데이터는 계정 경계로 보호한다.
- Alternatives considered:
  - 전 기능 로그인 필수: 초기 진입 전환율 저하
  - 전 기능 비로그인 허용: 개인화 데이터 보호 불가

## 4) 외부 데이터 의존 실패 모드
- Decision: 외부 소스 실패 시 마지막 유효값 + 갱신 시각 + 상태 배지 + 재시도 액션을 제공한다.
- Rationale: 실시간 데이터 결측은 불가피하므로 투명한 상태 노출이 신뢰 유지에 필수다.
- Alternatives considered:
  - 빈 화면 처리: 사용자 혼란 증가
  - 오류 은폐: 신뢰성/디버깅 저하

## 5) 테스트 전략
- Decision: Backend는 API 계약/권한/오류 경로 중심 pytest, Frontend는 핵심 사용자 여정 E2E 중심 테스트.
- Rationale: 본 기능은 API-UI 연계와 권한 분기가 많아 계약 테스트 + 사용자 여정 검증 조합이 효과적이다.
- Alternatives considered:
  - 단위 테스트 중심: 교차 흐름 리스크 탐지 한계
  - E2E만 사용: 실패 원인 지역화 어려움

## 6) 캐싱 전략
- Decision: 데이터 유형별 TTL 분리(일기 24h, 감성 30m, 시세/예측 단기 TTL)와 캐시 상태 표기를 적용한다.
- Rationale: 비용 제어와 신선도 균형을 동시에 충족하고, 사용자 신뢰를 위해 캐시 상태를 노출해야 한다.
- Alternatives considered:
  - 단일 TTL: 데이터 특성 반영 부족
  - 캐시 미사용: 비용/응답시간 악화

## 7) 개인정보/컴플라이언스
- Decision: GPS/정밀 위치 저장 금지, 민감 키는 서버 환경변수 전용, 로그에는 식별 민감정보 마스킹.
- Rationale: 헌법 및 README 보안 원칙과 직접 일치하며 법적/보안 리스크를 낮춘다.
- Alternatives considered:
  - 위치 기반 개인화 도입: 현재 요구 범위 밖 + 규제 리스크 증가

## 8) 결론
- 모든 기술 맥락의 확인 필요 항목을 해소했다.
- Phase 1 설계(데이터 모델/계약/퀵스타트) 진행 가능.
